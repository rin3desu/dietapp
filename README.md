2. 実装方針
2.1 開発コンセプト
「迷わないトレーニング」: ユーザーが登録した特定のジムにあるマシン構成に基づき、AIがその日のメニューを即座に提案する。

「データの統合管理」: 体重推移、食事ログ（写真）、トレーニング実績を一つのプラットフォームで完結させる。

「最新技術の社会実装」: 2026年時点の最新SDK（google-genai）とモデル（gemini-2.5-flash等）を導入し、実用的なAI体験を提供する。

2.2 重視する技術的観点
レスポンシブ・ファースト: CSS Gridとメディアクエリを活用し、PC（3列）とスマートフォン（1列）で最適化されたレイアウトを実現。

データの永続化: Renderの永続ディスク（Persistent Disk）を活用し、SQLiteデータベースがデプロイ後も保持される構成。

セキュアな設計: APIキーやシークレットキーを環境変数で管理し、ソースコードからの機密情報漏洩を防止。

4. 機能詳細設計
4.1 AIトレーニングメニュー推薦 (Core Feature)
機能概要: ユーザーが選択した「マイジム」に存在するマシンリストをプロンプトに埋め込み、AIが最適なメニューを生成。

使用モデル: gemini-2.5-flash（または gemini-3-flash-preview）。

プロンプト制御:

出力形式を「種目名・セット数・回数・アドバイス」のリスト形式に厳格に固定。

導入文や挨拶を排除し、トレーニング現場での視認性を最優先した簡潔なテキストを出力。

4.2 マイジム・マシン管理機能
拠点登録: 最大2件までのジムを登録可能。Leaflet.jsとOpenStreetMapを連携させ、緯度・経度ベースで位置情報を保持。

マシン管理: ジムごとに「チェストプレス（胸）」「ラットプルダウン（背中）」等のマシン名を部位カテゴリと共にDB登録。

4.3 記録管理機能
体重記録: 日次の記録に加え、前日比の自動計算および推移グラフ（Chart.js）の表示。

食事記録: 4つの時間帯（朝・昼・夕・間食）ごとの写真アップロードおよびログ機能。

トレーニング記録: 日付・部位・種目・重量・回数の詳細なセット記録。

6. インフラ・運用設計
6.1 デプロイ環境 (Render)
Web Service: Python 3.x環境での実行。

Environment Variables: GEMINI_API_KEY を含む機密情報の管理。

Persistent Disk: データベース（diet.db）およびアップロード画像の永続化。

6.2 エラーハンドリング
API制限 (429エラー): クォータ制限発生時のリトライ案内およびエラーログ出力。

モデル未検出 (404エラー): APIのバージョン更新に伴うモデル名の動的修正対応。

バリデーション: フォームの未入力項目、または対象部位のマシンが未登録時のフラッシュメッセージ表示。

7. 将来の展望
EdTechへの応用: 本アプリの学習ログ機能を応用し、大学受験生向けの「志望校別学習プランAI推薦システム」への横展開。

自動運転/地域支援: 地方在住の祖父母を支援する自動運転技術の研究背景を活かし、高齢者向けの健康維持・運動促進アプリとしての拡張。

分析高度化: 蓄積データから総負荷量（Volume）を算出し、筋肥大に最適な負荷を自動算出するアルゴリズムの搭載。
